# 수정지시서 002: @call 코드젠 수정 + validateCurrentUserType 제거

## 문제 1: guard-style @call이 2-value return을 무시

guard-style `@call` (result 없음)은 패키지 함수를 호출하여 에러 시 401을 반환한다. 그러나 모든 func spec 함수는 `(Response, error)` 2개 값을 반환하므로, 생성 코드가 컴파일되지 않는다.

### 현재 생성 코드 (오류)

```go
// @call auth.VerifyPassword(user.PasswordHash, request.Password)

if err = auth.VerifyPassword(auth.VerifyPasswordRequest{PasswordHash: user.PasswordHash, Password: password}); err != nil {
```

`auth.VerifyPassword`는 `(VerifyPasswordResponse, error)`를 반환하므로 `assignment mismatch: 1 variable but returns 2 values`.

### 올바른 생성 코드

```go
if _, err = auth.VerifyPassword(auth.VerifyPasswordRequest{PasswordHash: user.PasswordHash, Password: password}); err != nil {
```

### 수정

generator의 `@call` guard-style (Result == nil) 코드젠에서 `_, err :=` 패턴 사용.

## 문제 2: @call 구조체 리터럴의 필드명이 Arg.Field를 그대로 사용

`@call` 코드젠이 `Request` 구조체 리터럴을 생성할 때, SSaC Arg의 `Field` 이름을 구조체 필드명으로 사용한다. 그러나 Arg의 Field는 **소스 변수의 필드명**이지, **타겟 구조체의 필드명**이 아니다.

### 예시

```go
// @call IssueTokenResponse token = auth.IssueToken(user.ID, user.Email, user.Role)

// SSaC args:
//   Arg{Source: "user", Field: "ID"}
//   Arg{Source: "user", Field: "Email"}
//   Arg{Source: "user", Field: "Role"}

// 현재 생성 코드 (오류)
auth.IssueTokenRequest{ID: user.ID, Email: user.Email, Role: user.Role}

// IssueTokenRequest 실제 구조체:
// type IssueTokenRequest struct {
//     UserID int64   ← 첫 번째 필드
//     Email  string  ← 두 번째 필드
//     Role   string  ← 세 번째 필드
// }

// 올바른 생성 코드
auth.IssueTokenRequest{UserID: user.ID, Email: user.Email, Role: user.Role}
```

`ID` != `UserID` → 컴파일 에러.

### 원인

SSaC codegen이 arg를 구조체 리터럴로 변환할 때 `Arg.Field`를 구조체 필드명으로 사용한다. 하지만 args는 **위치 기반(positional)**이므로, i번째 arg는 Request 구조체의 i번째 필드에 매핑되어야 한다.

### 수정 방안

`@call` 코드젠 시 func spec의 Request 구조체 필드 목록을 참조하여 위치 기반 매핑:

```
arg[0] → RequestField[0].Name (UserID)  — value: user.ID
arg[1] → RequestField[1].Name (Email)   — value: user.Email
arg[2] → RequestField[2].Name (Role)    — value: user.Role
```

생성 코드:
```go
auth.IssueTokenRequest{UserID: user.ID, Email: user.Email, Role: user.Role}
```

SSaC generator가 이미 외부 검증(symbol table)에서 func spec을 로드하므로, 코드젠 시에도 이 정보를 활용할 수 있다.

func spec 접근이 어려우면 **대안**: 구조체 리터럴 대신 위치 기반 초기화 사용:
```go
auth.IssueTokenRequest{user.ID, user.Email, user.Role}
```
단, 이 방식은 필드 순서 변경에 취약하므로 named field가 선호됨.

## 문제 3: validateCurrentUserType 제거

### 현재

SSaC validator가 `currentUser` 소스를 사용하는 서비스 함수에 대해 `model/`에 `CurrentUser` 타입이 정의되어 있는지 검증한다.

```go
// validator.go
func validateCurrentUserType(sf parser.ServiceFunc, st *SymbolTable) []ValidationError {
    // ...
    if !st.HasCurrentUserType {
        // WARNING: "currentUser를 사용하지만 model/에 CurrentUser 타입이 정의되지 않았습니다"
    }
}
```

### 문제

CurrentUser는 **인증 인프라**이지 SSaC의 책임이 아니다:

- CurrentUser의 정의와 JWT claims 매핑은 **fullend.yaml**이 담당 (프로젝트 설정)
- fullend가 claims 설정으로 CurrentUser 타입 + 미들웨어를 **코드젠**
- SSaC는 `currentUser`가 예약 소스라는 것만 알면 되고, 타입 존재 여부는 **fullend crosscheck**가 검증
- `model/current_user.go`를 SSaC가 요구하는 것은 월권

### 수정

1. `validator.go`에서 `validateCurrentUserType` 함수 제거
2. `Validate` 함수에서 `validateCurrentUserType` 호출 제거
3. `symbol.go`에서 `HasCurrentUserType` 필드 및 관련 파싱 로직 제거
4. 관련 테스트 제거 (`TestValidateCurrentUserTypeMissing`, `TestValidateCurrentUserTypeExists`)

## 영향

- fullend dummy-lesson: `login.go`의 `@call auth.VerifyPassword` (문제1) 및 `@call auth.IssueToken` (문제2) 빌드 실패
- fullend 프로젝트에서 `model/current_user.go` 불필요 (fullend.yaml claims 설정으로 대체)
