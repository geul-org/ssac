# 수정지시서 003: Model 인터페이스 DDL 타입 반영 + JSON body 파싱 + authorize @id 안전처리 + 출력 폴더 구조 개선

## 1. Model 인터페이스 파라미터 타입이 전부 string

### 문제

`model_interface.go`의 `resolveParamType()`이 DDL 컬럼을 찾지 못하면 `"string"`으로 fallback한다.
실제로는 대부분의 ID 파라미터가 DDL에서 `BIGINT` (→ `int64`)인데, snake_case 변환 또는 테이블 탐색 실패로 string이 된다.

```go
// 현재 생성
type CourseModel interface {
    FindByID(courseID string) (*Course, error)
    Delete(courseID string) error
    Create(userID string, title string, description string, price string) (*Course, error)
}

// 기대
type CourseModel interface {
    FindByID(courseID int64) (*Course, error)
    Delete(courseID int64) error
    Create(userID int64, title string, description string, price int64) (*Course, error)
}
```

### 원인

`resolveParamType()`에서 `toSnakeCase(paramName)` → DDL 컬럼 조회 시:
1. `@model Course.FindByID`의 `@param CourseID` → `course_id` → `courses` 테이블의 `course_id` 컬럼은 보통 `id`로 정의됨 → 매칭 실패
2. `{Model}ID` 패턴을 인식하지 못함 — `CourseID`는 `courses.id` 컬럼에 매핑되어야 함
3. fallback `"string"` 적용

### 수정 제안

`resolveParamType()`에 다음 로직 추가:

```go
func resolveParamType(paramName string, modelName string, st *validator.SymbolTable) string {
    snakeName := toSnakeCase(paramName)

    // 1. 해당 모델의 테이블에서 직접 조회
    tableName := toSnakeCase(modelName) + "s" // Course → courses
    if table, ok := st.DDLTables[tableName]; ok {
        if col, ok := table.Columns[snakeName]; ok {
            return col.GoType
        }
    }

    // 2. {Model}ID 패턴 인식: CourseID → courses.id
    if strings.HasSuffix(paramName, "ID") {
        refModel := paramName[:len(paramName)-2] // CourseID → Course
        refTable := toSnakeCase(refModel) + "s"  // Course → courses
        if table, ok := st.DDLTables[refTable]; ok {
            if col, ok := table.Columns["id"]; ok {
                return col.GoType
            }
        }
    }

    // 3. 전체 테이블 순회 (기존 로직)
    for _, table := range st.DDLTables {
        if col, ok := table.Columns[snakeName]; ok {
            return col.GoType
        }
    }

    return "string" // 최종 fallback
}
```

### 영향

- Model 인터페이스의 모든 ID 파라미터가 `int64`로 정확히 생성됨
- oapi-codegen의 ServerInterface 타입과 일치
- 서비스 함수에서 path parameter로 받은 `int64` 값을 바로 모델에 전달 가능

## 2. Request body를 FormValue로 파싱

### 문제

현재 `generateExtractCode()`가 모든 `@param ... request` 파라미터를 `r.FormValue()`로 추출한다.
OpenAPI에서 `application/json` request body를 정의한 엔드포인트에서는 JSON decode가 필요하다.

```go
// 현재 생성
title := r.FormValue("Title")
description := r.FormValue("Description")
price, err := strconv.ParseInt(r.FormValue("Price"), 10, 64)

// 기대
var req struct {
    Title       string `json:"title"`
    Description string `json:"description"`
    Price       int64  `json:"price"`
}
if err := json.NewDecoder(r.Body).Decode(&req); err != nil {
    http.Error(w, "invalid request body", http.StatusBadRequest)
    return
}
title := req.Title
description := req.Description
price := req.Price
```

### 수정 제안

`@param ... request` 파라미터가 2개 이상이면 JSON body로 간주하고 struct decode를 생성한다.
(path parameter는 수정지시서002에서 이미 함수 인자로 분리되었으므로, 남는 request 파라미터는 body 필드다.)

**판별 기준:**
- path parameter에 해당하는 `@param`은 이미 함수 인자 → 제외
- 남은 `@param ... request`가 있으면 → JSON body struct 생성
- 1개만 남으면 → FormValue 유지 (쿼리 파라미터일 수 있음)

### 구현 위치

- `generator/generator.go` `generateExtractCode()` — JSON decode 코드 생성 분기 추가
- `generator/templates.go` — JSON request body decode 템플릿 추가

## 3. authorize @id에서 Go 예약어 충돌

### 문제

SSaC spec에서 `@id new`를 사용하면 생성 코드에 Go 예약어 `new`가 변수로 사용된다:

```go
// SSaC spec
// @sequence authorize
// @action create
// @resource course
// @id new

// 현재 생성 (컴파일 에러)
allowed, err := authz.Check(currentUser, "create", "course", new)
```

`new`는 Go 내장 함수이므로 변수로 사용하면 의미가 모호하다.

### 수정 제안

`resolveParamRef()`에서 Go 예약어/내장 함수 목록과 비교하여 충돌 시 접미사를 붙인다:

```go
var goReserved = map[string]bool{
    "new": true, "make": true, "len": true, "cap": true,
    "delete": true, "copy": true, "append": true, "close": true,
    "panic": true, "recover": true, "print": true, "println": true,
}

func resolveParamRef(raw string) string {
    // ... 기존 로직 ...
    result := lcFirst(raw)
    if goReserved[result] {
        return result + "Val"
    }
    return result
}
```

또는, `@id`가 리터럴 의미인 경우 (새로 생성하는 리소스에 대한 authorize) `nil`로 변환하는 것이 더 적절하다:

```go
// "new"는 "아직 존재하지 않는 리소스"를 의미하므로 nil이 자연스러움
if raw == "new" {
    return "nil"
}
```

**권장:** `@id new` → `nil` 변환. 리소스 생성 전 authorize에서 ID가 없는 것이 의미상 맞다.

## 4. 출력 폴더 구조 개선

### 현재 구조

```
outDir/
├── create_course.go     # package service
├── get_course.go
├── ...
└── model/
    └── models_gen.go    # package model
```

### 문제

- `service/` 폴더 없이 outDir 직하에 서비스 파일이 생성됨
- fullend가 glue 코드 생성 시 `backend/service/`에 넣으려면 outDir을 `backend/service`로 지정해야 함
- model은 `outDir/model/`에 생성되므로 service와 model의 상대 경로가 비일관적
- `internal/` 패키지 경계가 없어 외부 접근 제한 불가

### 수정 제안

Generator에 출력 기본 구조 옵션을 추가한다:

```go
// Generate에 옵션 추가
type GenerateOptions struct {
    OutDir     string
    UseInternal bool  // true면 internal/ 하위에 생성
}
```

`UseInternal = true`일 때:

```
outDir/
├── internal/
│   ├── service/
│   │   ├── create_course.go   # package service
│   │   └── get_course.go
│   └── model/
│       └── models_gen.go      # package model
```

`UseInternal = false`일 때 (기본, 하위 호환):

```
outDir/
├── create_course.go           # package service (기존 동작)
└── model/
    └── models_gen.go          # package model
```

### 대안

fullend가 outDir을 `backend/internal/service`, `backend/internal/model`로 각각 호출하는 방식도 가능하다.
이 경우 SSaC 수정 없이 fullend만 변경하면 된다.

**권장:** SSaC는 현행 유지, fullend가 outDir을 세분화하여 호출. SSaC의 책임 범위를 최소화한다.

| Model 인터페이스 DDL 타입 | **높음** | 전체 타입 시스템 불일치의 근본 원인 |
