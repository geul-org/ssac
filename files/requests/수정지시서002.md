## 선행 조건

## BUG005: sort 파라미터 SQL Injection (HIGH)

### 증상

`ParseQueryOpts`에서 allowlist 검증 후, `c.Query("sort")`로 재파싱하여 검증 없이 `opts.SortCol`을 덮어씀.

```go
// 생성되는 코드 (잘못됨)
opts := model.ParseQueryOpts(c, model.QueryOptsConfig{...})
if v := c.Query("sort"); v != "" {
    opts.SortCol = v  // ← allowlist 우회
}
```

### 수정

`ParseQueryOpts` 이후 sort/direction을 재파싱하는 코드 생성을 제거. `ParseQueryOpts`가 반환한 값을 그대로 사용.

### 대상 파일

- `go_params.go` (리팩토링 후) — `x-sort` 처리 시 재파싱 코드 생성 부분

### 검증

생성된 코드에서 `ParseQueryOpts` 이후 `c.Query("sort")` 또는 `c.Query("direction")`이 없어야 함.

## BUG006: password_hash JSON 노출 (HIGH)

### 증상

DDL의 `password_hash` 컬럼이 `json:"password_hash"` 태그로 생성되어 API 응답에 bcrypt 해시 노출.

```go
// 생성되는 코드 (잘못됨)
type User struct {
    PasswordHash string `json:"password_hash"`
}
```

### 수정

model 생성 시 컬럼명에 `password`, `secret`, `token_hash`, `hash` 패턴이 포함되면 `json:"-"` 태그 적용.

```go
// 기대 결과
type User struct {
    PasswordHash string `json:"-"`
}
```

### 대상 파일

- ssac model 코드젠 (DDL → Go struct 변환 부분)

### 검증

생성된 `types.go`에서 `PasswordHash` 필드의 json 태그가 `"-"`인지 확인.

## BUG007: authz.Check에 claims 미전달 (HIGH)

### 증상

`@auth` 시퀀스에서 `authz.Check` 호출 시 현재 유저 claims가 누락되어 소유권 검증 불가.

```go
// 생성되는 코드 (잘못됨)
authz.Check(authz.CheckRequest{
    Action:     "read",
    Resource:   "project",
    ResourceID: project.ID,
})
```

### 수정

`@auth` 코드 생성 시 `currentUser` (BearerAuth 미들웨어에서 주입)의 ID를 `CheckRequest.Claims`에 포함.

```go
// 기대 결과
authz.Check(authz.CheckRequest{
    Action:     "read",
    Resource:   "project",
    ResourceID: project.ID,
    Claims:     authz.Claims{UserID: currentUser.ID},
})
```

### 대상 파일

- `go_handler.go` (리팩토링 후) — `@auth` 시퀀스 코드 생성 부분

### 검증

`@auth`가 있는 모든 생성된 핸들러에서 `Claims` 필드가 포함되어 있는지 확인.

## BUG008: @call 에러 일괄 500 반환 (HIGH)

### 증상

`@call auth.VerifyPassword` 실패 시 비밀번호 불일치(401)가 아닌 500 Internal Server Error 반환.

```go
// 생성되는 코드 (잘못됨)
if err := auth.VerifyPassword(...); err != nil {
    c.JSON(http.StatusInternalServerError, gin.H{"error": "호출 실패"})
    return
}
```

### 수정

SSaC `@call` 문법에 에러 코드를 지정할 수 있도록 확장.

```
@call auth.VerifyPassword(hashedPassword, password) 401 "Invalid credentials"
```

문법: `@call pkg.Func(args...) [statusCode] ["message"]`
- statusCode 생략 시 기본 500 유지 (기존 동작 호환)
- statusCode 지정 시 해당 코드로 응답

```go
// 기대 결과
if err := auth.VerifyPassword(...); err != nil {
    c.JSON(http.StatusUnauthorized, gin.H{"error": "Invalid credentials"})
    return
}
```

### 대상 파일

- SSaC 파서 — `@call` 라인에서 statusCode, message 추출
- `go_handler.go` (리팩토링 후) — `@call` 에러 처리 코드 생성 부분

### 검증

1. statusCode 없는 기존 `@call` → 500 유지 (하위 호환)
2. `@call auth.VerifyPassword(...) 401 "Invalid credentials"` → 401 반환 확인
