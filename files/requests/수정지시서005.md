# 수정지시서005: @call 에러 코드 — 대상 함수의 `@error` 어노테이션 읽기

## 발신

fullend 프로젝트

## 문제

`@call` 시퀀스의 에러 핸들링이 무조건 `http.StatusInternalServerError` (500)을 반환한다. 인증 관련 호출(VerifyPassword 등)이 실패해도 500이 반환되어 API 의미가 잘못된다.

### 현재 생성 코드

```go
// login.ssac: @call auth.VerifyPassword({...})
if _, err = auth.VerifyPassword(...); err != nil {
    c.JSON(http.StatusInternalServerError, gin.H{"error": "호출 실패"})  // ← 항상 500
    return
}
```

### 기대 동작

```go
c.JSON(http.StatusUnauthorized, gin.H{"error": "호출 실패"})  // ← 401
```

## 설계

SSaC가 `@call` 대상 패키지를 이미 Go AST로 파싱하고 있다(`loadPackageGoInterfaces`). 함수의 doc comment에 `// @error NNN`을 선언하면, SSaC가 이를 읽어 `@call` 에러 코드 기본값으로 사용한다.

### 우선순위

```
1. .ssac 파일에서 명시: @call auth.VerifyPassword({...}) 500  → 500
2. 대상 함수의 @error 어노테이션: // @error 401                → 401
3. 둘 다 없으면 기본값                                          → 500
```

### fullend 측 변경 (SSaC 범위 밖)

```go
// pkg/auth/verify_password.go
// @func verifyPassword
// @error 401                  ← 추가
// @description 저장된 해시와 평문 비밀번호가 일치하는지 검증한다

func VerifyPassword(req VerifyPasswordRequest) (VerifyPasswordResponse, error) { ... }
```

fullend가 제공하는 pkg/ 함수가 자체적으로 실패 시 적절한 HTTP 상태 코드를 SSOT로 선언한다. 이 변경은 fullend 프로젝트에서 수행하며, SSaC 수정 범위가 아니다.

## SSaC 수정 사항

### 1. `MethodInfo`에 `ErrStatus` 필드 추가

```go
// validator/symbol.go
type MethodInfo struct {
    Cardinality string
    Params      []string
    ParamTypes  map[string]string
    ErrStatus   int               // ← 추가: @error 어노테이션 값 (0이면 미지정)
}
```

### 2. `loadPackageGoInterfaces`에서 `@error` 파싱

현재 `loadPackageGoInterfaces`는 두 단계로 동작한다:
- 1차: `{Method}Request` struct 수집
- 2차: interface 파싱 → `MethodInfo`에 `ParamTypes` 매칭

**문제**: `@call` 대상 함수(auth.VerifyPassword 등)는 **standalone function**이지 interface method가 아니다. 현재 2차 스캔에서 interface만 찾으므로 standalone function은 `st.Models`에 등록되지 않는다.

**해결**: 3차 스캔을 추가하여 standalone function도 등록한다.

```go
// validator/symbol.go — loadPackageGoInterfaces 끝부분에 추가

// 3차: standalone function 파싱 (@call 대상)
// interface에 속하지 않는 func Xxx(req XxxRequest) (XxxResponse, error) 패턴
for _, entry := range entries {
    if entry.IsDir() || !strings.HasSuffix(entry.Name(), ".go") {
        continue
    }
    f, err := parser.ParseFile(fset, filepath.Join(dir, entry.Name()), nil, parser.ParseComments)
    if err != nil {
        continue
    }
    for _, decl := range f.Decls {
        fd, ok := decl.(*ast.FuncDecl)
        if !ok || fd.Recv != nil {
            continue // method가 아닌 standalone function만
        }
        funcName := fd.Name.Name
        reqStructName := funcName + "Request"
        if _, ok := requestStructs[reqStructName]; !ok {
            continue // {Func}Request struct가 없으면 스킵
        }

        // @error 어노테이션 파싱
        errStatus := 0
        if fd.Doc != nil {
            for _, comment := range fd.Doc.List {
                text := strings.TrimSpace(strings.TrimPrefix(comment.Text, "//"))
                if strings.HasPrefix(text, "@error ") {
                    if code, err := strconv.Atoi(strings.TrimSpace(text[7:])); err == nil {
                        errStatus = code
                    }
                }
            }
        }

        // standalone function을 "pkgName.Func" 모델 키로 등록
        // 기존 interface 등록과 동일한 st.Models 맵 사용
        modelKey := pkgName + "._func"  // 예: "auth._func"
        ms, exists := st.Models[modelKey]
        if !exists {
            ms = ModelSymbol{Methods: make(map[string]MethodInfo)}
        }
        mi := MethodInfo{
            ParamTypes: requestStructs[reqStructName],
            ErrStatus:  errStatus,
        }
        ms.Methods[funcName] = mi
        st.Models[modelKey] = ms
    }
}
```

`_func`를 모델 키로 사용하는 이유: 기존 interface 모델(`auth.Session` 등)과 이름 충돌을 방지하면서, 기존 `validateCallInputTypes`(validator.go:604)의 `strings.HasPrefix(modelKey, pkgName+".")` 조건과 호환된다.

### 3. `go_helpers.go`에서 `@error` 코드 참조

`buildTemplateData`의 `@call` ErrStatus 결정 로직을 수정:

```go
// generator/go_helpers.go — buildTemplateData 내부, 현재 78-82행
if seq.ErrStatus != 0 {
    d.ErrStatus = httpStatusConst(seq.ErrStatus)
} else if st != nil {
    // SymbolTable에서 @error 어노테이션 조회
    errCode := lookupCallErrStatus(st, seq.Model)
    if errCode != 0 {
        d.ErrStatus = httpStatusConst(errCode)
    } else {
        d.ErrStatus = "http.StatusInternalServerError"
    }
} else {
    d.ErrStatus = "http.StatusInternalServerError"
}
```

`lookupCallErrStatus` 헬퍼:

```go
func lookupCallErrStatus(st *validator.SymbolTable, model string) int {
    parts := strings.SplitN(model, ".", 2)
    if len(parts) < 2 {
        return 0
    }
    pkgName, funcName := parts[0], parts[1]
    for modelKey, ms := range st.Models {
        if !strings.HasPrefix(modelKey, pkgName+".") {
            continue
        }
        if mi, ok := ms.Methods[funcName]; ok {
            return mi.ErrStatus
        }
    }
    return 0
}
```

### 4. `buildTemplateData`에 `st` 파라미터 전달

현재 `buildTemplateData` 시그니처에 `st *validator.SymbolTable`이 이미 포함되어 있다(go_helpers.go:64). 추가 변경 불필요.

## 변경 파일

| 파일 | 변경 |
|---|---|
| `validator/symbol.go` | `MethodInfo.ErrStatus` 필드 추가 + `loadPackageGoInterfaces` 3차 standalone function 파싱 + `@error` 어노테이션 읽기 |
| `generator/go_helpers.go` | `@call` ErrStatus에 SymbolTable `@error` 조회 로직 추가 + `lookupCallErrStatus` 헬퍼 |
| `generator/go_handler_test.go` | `@error 401` 함수 호출 시 기본 401 생성 테스트 |
| `validator/symbol_test.go` | `@error` 어노테이션 파싱 테스트 |

## 검증

```bash
go test ./validator/... ./generator/...
```

1. `@call auth.VerifyPassword({...})` + 대상 함수에 `@error 401` → 기본 401 생성
2. `@call auth.VerifyPassword({...}) 500` + 대상 함수에 `@error 401` → .ssac 명시값 500 우선
3. `@call billing.Charge({...})` + `@error` 어노테이션 없음 → 기본 500 유지
4. 기존 테스트 전체 통과

## 부수 효과

이 변경으로 `validateCallInputTypes`(validator.go:604-612)도 자동으로 개선된다. 현재 standalone function은 `st.Models`에 등록되지 않아 타입 검증이 스킵되지만, 3차 스캔으로 등록되면 `@call` 입력 타입 검증도 동작하게 된다.
