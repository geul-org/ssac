# 수정지시서 006: 005 후속 — Generate() 코드젠 잔여 불일치 4건

## 배경

수정지시서005 적용 후 `fullend gen`으로 dummy-lesson을 재생성한 결과, Generate()가 생성하는 서비스 코드에 4건의 불일치가 남아 있다.

---

## 1. FindByID에 opts가 잘못 전달됨

### 문제

005에서 QueryOpts 전달을 추가했으나, `FindByID` 같은 단건 조회에도 opts가 전달된다:

```go
// get_course.go (Generate 생성)
opts := QueryOpts{}
course, err := s.courseModel.FindByID(courseID, opts)  // ← FindByID는 opts 안 받음
```

모델 인터페이스:
```go
FindByID(courseID int64) (*Course, error)  // 1개 파라미터
```

### 원인

005 수정 시 해당 서비스 함수에 `HasQueryOpts()=true`이면 함수 전체에 `opts := QueryOpts{}`를 넣고 모든 모델 호출에 opts를 추가한 것으로 보인다. 그러나 opts는 **List 계열 메서드**에만 전달해야 한다.

### 수정

opts 전달은 모델 메서드 단위로 판단해야 한다:
- `List`, `ListBy*` → opts 전달
- `FindByID`, `FindBy*`, `Create`, `Update`, `Delete`, `Publish` → opts 미전달

```go
// buildParamArgs에서 opts 추가 조건
if isListMethod(seq.Model) && hasQueryOpts {
    args = append(args, "opts")
}
```

`isListMethod`는 메서드명이 `List` 또는 `ListBy`로 시작하는지 확인.

### 영향 파일

- `get_course.go` — `FindByID(courseID, opts)` → `FindByID(courseID)`
- 기타 서비스에서 FindBy*, Create, Update, Delete에 opts가 전달되는 경우 모두 해당

## 2. ListLessons에 opts가 누락됨

### 문제

`list_lessons.go`는 opts를 전달하지 않는다:

```go
// list_lessons.go (Generate 생성)
func (s *Server) ListLessons(w http.ResponseWriter, r *http.Request, courseID int64) {
    lessons, err := s.lessonModel.ListByCourse(courseID)  // ← opts 누락
```

모델 인터페이스:
```go
ListByCourse(courseID int64, opts QueryOpts) ([]Lesson, int, error)  // 2개 파라미터
```

비교: `list_reviews.go`는 정상:
```go
opts := QueryOpts{}
reviews, total, err := s.reviewModel.ListByCourse(courseID, opts)  // ✓ 정상
```

### 원인

005 수정이 일부 서비스에만 적용된 것으로 보인다. ListLessons의 OpenAPI에 x-pagination 등이 없거나 `HasQueryOpts()` 판단이 잘못되었을 수 있다.

### 수정

ListByCourse 호출의 opts 전달은 **OpenAPI 엔드포인트 단위가 아니라 모델 인터페이스 단위**로 판단해야 한다:
- `LessonModel.ListByCourse`가 `(courseID, opts)` 시그니처이면, 호출하는 모든 서비스에서 opts 전달

또는 더 단순하게: `isListMethod()` 이름이면 항상 opts를 전달한다 (이슈 1의 수정과 통일).

### 영향 파일

- `list_lessons.go` — `ListByCourse(courseID)` → `ListByCourse(courseID, opts)`
- List 계열을 호출하는 모든 서비스에서 일관 적용

## 3. 리터럴 파라미터의 인터페이스 이름이 잘못됨

### 문제

005에서 리터럴 `@param`을 모델 인터페이스에 포함하도록 수정했으나, 파라미터 이름이 부정확하다.

**UserModel.Create:**

SSaC spec:
```
@param Email request
@param hashedPassword       ← @func 결과
@param Name request
@param "student"            ← 리터럴 (role)
```

생성된 인터페이스:
```go
Create(email string, hashedPassword string, name string, passwordHash string) (*User, error)
//                                                       ^^^^^^^^^^^^ "student"인데 passwordHash로 명명
```

4번째 파라미터는 `"student"` 리터럴이므로 `role string`이어야 한다.

**PaymentModel.Create:**

SSaC spec:
```
@param UserID currentUser
@param enrollment.ID        ← 이전 결과
@param course.Price         ← 이전 결과
@param PaymentMethod request
@param "pending"            ← 리터럴 (status)
```

생성된 인터페이스:
```go
Create(userID int64, enrollmentID int64, coursePrice int64, paymentMethod string, method string) (*Payment, error)
//                                                                                ^^^^^^ "pending"인데 method로 명명
```

5번째 파라미터는 `"pending"` 리터럴이므로 `status string`이어야 한다.

### 원인

리터럴 `@param`의 이름 결정 로직에서 리터럴 값이 아니라 DDL 컬럼명을 매칭해야 하는데, 잘못된 컬럼에 매칭되거나 fallback 이름이 부정확하다.

### 수정 제안

리터럴 파라미터의 이름 결정:

1. DDL에서 해당 테이블의 컬럼 중 리터럴 값이 들어갈 수 있는 컬럼을 찾는다
   - `"student"` → `users` 테이블에서 `role` 컬럼 (VARCHAR, enum 등)
   - `"pending"` → `payments` 테이블에서 `status` 컬럼

2. DDL 매칭이 불가능하면 파라미터 순서(position) 기반으로 DDL 컬럼 순서에서 대응하는 이름을 사용한다

3. 최후 fallback: `param{N}` (예: `param4`)

### 영향 파일

- `generator/go_target.go` (또는 `model_interface.go`) — 리터럴 @param 이름 결정 로직 수정

## 4. total 변수가 response JSON에 미포함

### 문제

005에서 List 3-tuple 반환 (`items, total, err`)을 적용했으나, response JSON에 `total`이 포함되지 않는다:

```go
// get_course.go
lessons, total, err := s.lessonModel.ListByCourse(courseID, opts)
// ...
json.NewEncoder(w).Encode(map[string]interface{}{
    "course":  course,
    "lessons": lessons,
    // ← total 누락
})
```

```go
// list_reviews.go
reviews, total, err := s.reviewModel.ListByCourse(courseID, opts)
json.NewEncoder(w).Encode(map[string]interface{}{
    "reviews": reviews,
    // ← total 누락
})
```

Go 컴파일러가 `total` 미사용 에러를 발생시킨다.

### 수정

response json 시퀀스의 템플릿에서, List 3-tuple 결과가 있으면 `total`을 JSON에 포함한다:

```go
// 수정 후 생성
json.NewEncoder(w).Encode(map[string]interface{}{
    "reviews": reviews,
    "total":   total,
})
```

또는 `total`이 response에 불필요하면 `_`로 무시:
```go
reviews, _, err := s.reviewModel.ListByCourse(courseID, opts)
```

**권장**: total을 포함한다. 프론트엔드 페이지네이션에 필요하다.

### 영향 파일

- `generator/go_target.go` 또는 `generator/go_templates.go` — response json 템플릿에 total 조건부 추가

## 5. sqlc 쿼리 이름에 모델명 접두사 지원

### 문제

sqlc는 모든 쿼리 파일을 하나의 Go 패키지로 합친다. 따라서 여러 파일에 `FindByID`, `Create` 등 동일한 쿼리 이름이 있으면 충돌한다:

```
db/queries/course.sql:      -- name: FindByID :one
db/queries/enrollments.sql:  -- name: FindByID :one
db/queries/lessons.sql:      -- name: FindByID :one
...
```

```
$ sqlc compile
duplicate query name: FindByID
duplicate query name: Create
...
```

### 해결 방향

쿼리 파일의 `-- name:` 에 모델명 접두사를 붙인다:

```sql
-- course.sql
-- name: CourseFindByID :one
-- name: CourseList :many
-- name: CourseCreate :one

-- enrollments.sql
-- name: EnrollmentFindByID :one
-- name: EnrollmentCreate :one
```

### ssac 수정 필요

ssac의 `@model Course.FindByID` 검증 시, 쿼리 파일에서 매칭하는 이름 규칙을 변경해야 한다:

- 현재: `@model Course.FindByID` → 쿼리 이름 `FindByID` 검색
- 수정: `@model Course.FindByID` → 쿼리 이름 `CourseFindByID` 검색 (모델명 + 메서드명)

`validator/symbol_table.go`의 `LoadSymbolTable()`에서 sqlc 쿼리를 로드할 때, `-- name: CourseFindByID`를 `Course` + `FindByID`로 분리하여 SymbolTable에 등록한다.

분리 규칙: 쿼리 파일명(단수형 PascalCase)이 접두사이면 분리. 예:
- 파일 `course.sql`, 쿼리 `CourseFindByID` → Model=`Course`, Method=`FindByID`
- 파일 `enrollments.sql`, 쿼리 `EnrollmentFindByID` → Model=`Enrollment`, Method=`FindByID`

### 영향 파일

- `validator/symbol_table.go` — 쿼리 이름 파싱 시 모델명 접두사 분리 로직 추가

## 6. 서비스 파일의 패키지/디렉토리 불일치

### 문제

ssac generator가 서비스 파일을 `internal/service/` 디렉토리에 생성하면서 `package internal`로 선언한다:

```
internal/service/create_course.go   → package internal  ← 불일치!
internal/service/list_courses.go    → package internal  ← 불일치!
```

Go에서는 디렉토리 = 패키지이므로, `internal/service/` 안의 파일은 `package service`여야 하거나, 서비스 파일을 `internal/` 에 직접 놓아야 한다.

### 해결

서비스 파일을 `internal/` 디렉토리에 직접 생성한다 (service 서브디렉토리 없이). `server.go`와 같은 패키지에 있어야 `Server` 구조체의 메서드로 정의할 수 있다.

```
internal/
├── server.go            ← package internal
├── create_course.go     ← package internal (같은 디렉토리)
├── list_courses.go      ← package internal
└── model/
    └── models_gen.go    ← package model
```

### 영향 파일

- `generator/go_target.go` — 서비스 파일 출력 경로를 `internal/service/` → `internal/` 로 변경

---

## 7. server.go가 model 타입을 패키지 한정자 없이 참조

### 문제

`server.go`는 `package internal`에 속하는데, Model 인터페이스(`CourseModel`, `EnrollmentModel` 등)는 `package model`에 정의되어 있다:

```go
// internal/server.go — package internal
type Server struct {
    courseModel CourseModel           // ← undefined: CourseModel
    enrollmentModel EnrollmentModel  // ← undefined: EnrollmentModel
    ...
}
```

`CourseModel`은 `model.CourseModel`로 참조해야 한다.

### 해결

`server.go` 생성 시 model 패키지를 import하고 타입에 `model.` 접두사를 붙인다:

```go
package internal

import "dummy-lesson/backend/internal/model"

type Server struct {
    courseModel model.CourseModel
    enrollmentModel model.EnrollmentModel
    lessonModel model.LessonModel
    paymentModel model.PaymentModel
    reviewModel model.ReviewModel
    userModel model.UserModel
}
```

서비스 파일에서 Model 메서드의 반환 타입(`*Course`, `[]Lesson` 등)을 사용할 때도 `model.Course`, `model.Lesson` 등으로 한정해야 한다.

### 영향 파일

- `generator/go_target.go` 또는 `generator/go_templates.go` — server.go 템플릿에 model import 추가, 타입에 `model.` 접두사
