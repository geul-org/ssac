# 수정지시서 005: Generate()와 GenerateModelInterfaces() 시그니처 불일치 수정

## 배경

fullend gen으로 dummy-lesson을 생성한 결과, SSaC `Generate()`가 생성하는 서비스 함수 코드와 `GenerateModelInterfaces()`가 생성하는 모델 인터페이스 사이에 메서드 시그니처가 일치하지 않는다.
Go 컴파일이 실패한다.

---

## 1. List 계열 메서드에 QueryOpts가 전달되지 않음

### 문제

`GenerateModelInterfaces()`는 OpenAPI x-pagination/x-sort/x-filter/x-include가 있으면 `QueryOpts` 파라미터를 인터페이스에 추가한다:

```go
// models_gen.go (GenerateModelInterfaces 생성)
type CourseModel interface {
    List(opts QueryOpts) ([]Course, int, error)
}
type LessonModel interface {
    ListByCourse(courseID int64, opts QueryOpts) ([]Lesson, int, error)
}
```

그러나 `Generate()`는 해당 모델 메서드 호출에 QueryOpts를 전달하지 않는다:

```go
// list_courses.go (Generate 생성)
courses, err := courseModel.List()           // ← 0개 인자, 인터페이스는 1개
```

```go
// list_lessons.go (Generate 생성)
lessons, err := lessonModel.ListByCourse()   // ← 0개 인자, 인터페이스는 2개
```

### 원인

`generator.go`의 `buildParamArgs()`가 `@param` 태그의 명시적 파라미터만 수집한다.
QueryOpts는 `@param`으로 명시되지 않고, x- 확장에서 암묵적으로 파생되므로 `buildParamArgs()`에 포함되지 않는다.

### 수정 제안

`buildParamArgs()` (또는 `buildTemplateData()`)에서 해당 operation에 query options가 있으면 QueryOpts 인자를 자동 추가한다.

```go
func buildParamArgs(seq parser.Sequence, sf parser.ServiceFunc, st *validator.SymbolTable) string {
    args := // ... 기존 로직 ...

    // QueryOpts 자동 추가
    if st != nil {
        if op, ok := st.Operations[sf.Name]; ok && op.HasQueryOpts() {
            args = append(args, "opts")
        }
    }

    return strings.Join(args, ", ")
}
```

동시에 함수 본문 상단에 QueryOpts 구성 코드를 생성한다:

```go
// 생성할 코드
opts := QueryOpts{
    Limit:   limit,
    Offset:  offset,
    SortCol: sortBy,
    SortDir: sortDir,
    Filters: filters,
}
```

이 값들은 `r.URL.Query()`에서 추출한다. oapi-codegen의 `Params` struct가 있다면 그것을 사용한다.

**단순화 옵션:** QueryOpts 구성이 복잡하면, 우선 빈 QueryOpts를 전달하는 것으로 시작한다:

```go
courses, err := courseModel.List(QueryOpts{})
```

이렇게 하면 최소한 컴파일이 된다. 실제 query parameter 바인딩은 후속 작업으로 둘 수 있다.

### 영향 파일

- `list_courses.go` — `courseModel.List()` → `courseModel.List(opts)`
- `list_lessons.go` — `lessonModel.ListByCourse()` → `lessonModel.ListByCourse(courseID, opts)`
- `list_reviews.go` — `reviewModel.ListByCourse()` → `reviewModel.ListByCourse(courseID, opts)`
- `list_my_enrollments.go` — `enrollmentModel.ListByUser()` → `enrollmentModel.ListByUser(userID, opts)`
- `list_my_payments.go` — `paymentModel.ListByUser()` → `paymentModel.ListByUser(userID, opts)`

## 2. 파생 파라미터/리터럴이 모델 인터페이스에 누락

### 문제

SSaC spec에서 `@param` 소스가 `request`가 아닌 경우 (이전 결과 참조, 리터럴 문자열), `GenerateModelInterfaces()`가 해당 파라미터를 인터페이스에 포함하지 않는다.

**Payment.Create 예시:**

SSaC spec:
```
@model Payment.Create
@param UserID currentUser
@param enrollment.ID           ← 이전 결과 참조
@param course.Price            ← 이전 결과 참조
@param PaymentMethod request
@param "pending"               ← 리터럴
@result payment Payment
```

Generate() 생성 코드:
```go
payment, err := paymentModel.Create(currentUser.UserID, enrollment.ID, course.Price, paymentMethod, "pending")
// 5개 인자
```

GenerateModelInterfaces() 생성 인터페이스:
```go
type PaymentModel interface {
    Create(userID int64, paymentMethod string) (*Payment, error)
    // 2개 인자만 — enrollment.ID, course.Price, "pending" 누락
}
```

**User.Create 예시:**

SSaC spec:
```
@model User.Create
@param Email request
@param hashedPassword          ← 이전 @func 결과
@param Name request
@param "student"               ← 리터럴
@result user User
```

Generate() 생성 코드:
```go
user, err := userModel.Create(email, hashedPassword, name, "student")
// 4개 인자
```

GenerateModelInterfaces() 생성 인터페이스:
```go
type UserModel interface {
    Create(email string, hashedPassword string, name string) (*User, error)
    // 3개 인자만 — "student" 누락
}
```

### 원인

`model_interface.go`의 `collectModelUsages()` 또는 `deriveInterfaces()`가 `@param` 소스를 분석할 때:
- `request` 소스 → 파라미터 포함
- `currentUser` 소스 → 파라미터 포함
- 이전 결과 참조 (`enrollment.ID`, `course.Price`) → **누락**
- 리터럴 문자열 (`"pending"`, `"student"`) → **누락**

### 수정 제안

모든 `@param` 소스를 인터페이스 파라미터에 포함한다. 소스별 타입 결정:

| 소스 | 타입 결정 방법 |
|---|---|
| `request` | DDL 컬럼 타입 조회 (기존) |
| `currentUser` | `int64` (UserID) 또는 DDL 조회 |
| 이전 결과 참조 (`enrollment.ID`) | `enrollment`의 result 타입에서 ID 필드 → DDL 조회 |
| 리터럴 문자열 (`"pending"`) | `string` |
| 이전 @func 결과 (`hashedPassword`) | 이전 @result 타입 (`string`) |

**단순화 옵션:** 모든 파라미터 소스를 무조건 인터페이스에 포함하고, 타입을 결정할 수 없으면 해당 `@result` 타입 또는 `interface{}` fallback.

```go
// 수정 후 기대 결과
type PaymentModel interface {
    Create(userID int64, enrollmentID int64, price int64, paymentMethod string, status string) (*Payment, error)
}

type UserModel interface {
    Create(email string, hashedPassword string, name string, role string) (*User, error)
}
```

### 구현 위치

- `generator/go_target.go` (또는 이전 `model_interface.go`) `collectModelUsages()` — 모든 @param 소스 수집
- `generator/go_target.go` `deriveInterfaces()` — 파생 파라미터/리터럴의 타입 결정

## 3. List 계열 반환값 불일치 (Generate vs ModelInterfaces)

### 문제

`GenerateModelInterfaces()`는 `"many"` cardinality + QueryOpts일 때 `([]Type, int, error)` (count 포함 3-tuple) 반환을 생성한다.
그러나 `Generate()`는 `@result courses []Course`를 보고 2-tuple로 생성한다:

```go
// Generate 생성
courses, err := courseModel.List(opts)  // 2-tuple

// 인터페이스
List(opts QueryOpts) ([]Course, int, error)  // 3-tuple
```

### 수정 제안

`Generate()`에서 해당 operation에 QueryOpts가 있으면 count 변수를 추가한다:

```go
// 수정 후 생성
courses, total, err := courseModel.List(opts)
```

### 구현 위치

- `generator/go_target.go` `buildTemplateData()` 또는 get/post 시퀀스 템플릿
- `generator/go_templates.go` — get 템플릿에 count 변수 조건부 추가
