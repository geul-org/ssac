# 수정지시서 001: sqlFileToModel 단수화 + 심볼릭 선언 확장

## 증상

`courses.sql` 파일이 모델명 `Course`가 아닌 `Cour`로 변환된다.

```
입력: courses.sql
기대: Course
실제: Cour
```

결과적으로 `@model Course.FindByID` 등 모든 Course 모델 참조가 "모델을 찾을 수 없습니다" 에러를 발생시킨다.

## 원인

`validator/symbol.go:184-189` `sqlFileToModel` 함수의 단수화 규칙 순서 문제.

```go
if strings.HasSuffix(name, "ies") {
    name = name[:len(name)-3] + "y"
} else if strings.HasSuffix(name, "ses") || strings.HasSuffix(name, "xes") {
    name = name[:len(name)-2]       // ← 여기서 걸림
} else if strings.HasSuffix(name, "s") {
    name = name[:len(name)-1]
}
```

`courses`는 `ses`로 끝나므로 (`cour**ses**`) 두 번째 규칙에 매칭된다.
`es`를 제거하면 `cour`가 되고, PascalCase 변환 후 `Cour`가 된다.

## 영향받는 단어 패턴

`ses`로 끝나는 모든 복수형 테이블명:
- `courses` → `Cour` (기대: `Course`)
- `resources` → `Resour` (기대: `Resource`)
- `responses` → `Respon` (기대: `Response`)
- `expenses` → `Expen` (기대: `Expense`)

`ses` 규칙이 의도한 대상은 `addresses` → `address`, `buses` → `bus` 같은 경우다.

## 수정 제안

`ses` 규칙 적용 전 원형이 유효한지 확인하거나, 규칙 순서를 조정한다.

```go
// 방법 1: ses 규칙을 더 엄격하게 (자음+ses 패턴만)
// addresses → addres → address (ss+es)
// buses → bu → bus (s+es)
// courses → 해당 안됨 (rses가 아닌 일반 s)

// 방법 2: 알려진 예외 처리
// "ses"로 끝나되 실제로는 단순 s 복수형인 경우 마지막 s만 제거

// 방법 3: 가장 단순한 수정
if strings.HasSuffix(name, "ies") {
    name = name[:len(name)-3] + "y"
} else if strings.HasSuffix(name, "sses") || strings.HasSuffix(name, "xes") {
    name = name[:len(name)-2]
} else if strings.HasSuffix(name, "s") {
    name = name[:len(name)-1]
}
```

방법 3이 가장 안전하다: `sses` (addresses, classes)와 `xes` (boxes, indexes)만 `es` 제거하고, 나머지는 `s`만 제거.

## 재현

```bash
# dummy-lesson 프로젝트에서 courses.sql → Course 모델 참조 시
ssac validate ~/.clari/repos/fullend/specs/dummy-lesson/
```

# 추가 수정: 심볼릭 선언을 통한 WARNING 억제

fullend 교차 검증에서 발생하는 WARNING 중 **의도적인 설계 선택**인 경우가 있다.
이를 외부 suppress 파일이 아닌 **SSOT 원천 내 심볼릭 선언**으로 해결하고자 한다.

## 1. `@dto` 태그 — DDL 테이블 없는 모델 타입 선언

### 문제

`Token`, `Refund` 같은 타입은 DDL 테이블이 없는 순수 DTO다.
SSaC `@result` 에서 이 타입을 참조하면 fullend가 "DDL 테이블을 찾을 수 없습니다" WARNING을 발생시킨다.

### 제안

model 파일에서 `@dto` 주석 태그로 선언:

```go
// @dto
type Token struct {
    AccessToken string
    ExpiresAt   string
}
```

SSaC validator가 `@dto` 태그가 달린 타입을 SymbolTable에 DTO로 등록하면,
fullend는 해당 타입에 대해 DDL 테이블 매칭을 건너뛸 수 있다.

### 구현 위치

- `parser/` — model 파일 파싱 시 `@dto` 주석 감지
- `validator/symbol.go` — SymbolTable에 `DTOs map[string]bool` 또는 기존 구조에 플래그 추가

## 2. `-> column` 구문 — SSaC @param의 명시적 DDL 컬럼 매핑

### 문제

SSaC `@param CourseID request`에서 `CourseID`를 DDL 컬럼으로 변환할 때
`pascal_to_snake("CourseID")` → `course_id`로 추론한다.
하지만 `{Model}ID` 패턴이 아닌 경우나 컬럼명이 다른 경우 매칭이 실패한다.

### 제안

```go
// @param PaymentMethod request -> method
```

`-> method`가 있으면 `PaymentMethod`를 DDL의 `method` 컬럼에 매핑한다.
없으면 기존 자동 변환 로직 유지 (하위 호환).

### 구현 위치

- `parser/` — `@param` 라인에서 `->` 뒤 컬럼명 파싱
- `validator/` — 파싱된 매핑 정보를 SymbolTable 또는 ServiceFunc에 포함
- fullend `crosscheck/ssac_ddl.go` — 매핑 정보가 있으면 그 컬럼명으로 검증

## 우선순위

| 항목 | 긴급도 | 이유 |
|---|---|---|
| sqlFileToModel 단수화 | **높음** | 복수형 테이블명 사용 시 모델 참조 실패 (ERROR) |
| @dto 태그 | 중간 | WARNING 억제, DTO 의도 명시 |
| -> column 매핑 | 낮음 | 현재 자동 추론이 대부분 동작, edge case용 |
